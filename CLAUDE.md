## route
- always read `coding-style.md` before you code
- always check `coding-style.md` to see your code obey those rules

## rules
- answer in one-sentence, unless i ask you not to
- Don't over-engineer anything. Find existing mechanism that WordPress has before inventing anything. Don't go too far from what wordpress have.
- we value native-wordpress over general mechanism but fragile
- design architecture should be auditable. Don't use class unless you have to.
- everything should be self-documented, no guessing from 1st glance. file names, function names, variable names, directory names, asset slugs — all should tell the reader what they are before any docblock or doc page is opened.
- as lightweight as possible
